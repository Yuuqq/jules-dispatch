## Plan

### Step 1: Create `src/errors.ts` (ERR-01)

New module with these exports:

```typescript
export interface ErrorContext {
  hint?: string;
  docsUrl?: string;
}

export interface TranslatedError {
  problem: string;
  cause: string;
  fix: string;
  code: string;
  context?: ErrorContext;
}

export function translateError(err: unknown): TranslatedError
```

**Translation logic** (maps by detecting error characteristics):

| Detection | problem | cause | fix | code |
|-----------|---------|-------|-----|------|
| `err.status === 401` or `err.status === 403` | Authentication failed | API key is invalid, expired, or missing | Run `jules-dispatch init` or check `JULES_API_KEY` in `.env` | `AUTH_FAILED` |
| `err.status === 404` | Resource not found | The session or source ID does not exist | Check the ID and try again | `NOT_FOUND` |
| `err.status === 429` | Rate limited | Too many requests to Jules API | Wait a moment and retry. Reduce `--parallel` for batch commands | `RATE_LIMITED` |
| `err.status >= 500` | Jules API server error | The Jules service returned an internal error | Retry in a few minutes. Check status.google.com for outages | `SERVER_ERROR` |
| `err instanceof TypeError` + network pattern | Network error | Cannot reach the Jules API server | Check internet connection. If behind a proxy, set `HTTPS_PROXY` | `NETWORK_ERROR` |
| message contains `JULES_API_KEY` | API key not configured | No Jules API key was found | Set `JULES_API_KEY` in `.env` or pass `--api-key` | `AUTH_MISSING` |
| message contains `YAML` or `JSON` or `title` or `prompt` | Task file validation failed | The task file has invalid or missing fields | Fix the task file. Run `jules-dispatch doctor --task-file <path>` to validate | `VALIDATION` |
| fallback | Unexpected error | An unknown error occurred | Run with `--verbose` for details. File an issue at github.com/... | `UNKNOWN` |

**Key design decisions:**
- Detection is pattern-based (status code, error type, message content) — no new error classes needed
- `translateError` is a pure function — testable without mocks
- Reuses the same HTTP status detection already in `mcp-helpers.ts:computeRecoveryHint` — can be consolidated later

### Step 2: Enhance `emitError()` in `src/output.ts` (ERR-02)

Extend the existing signature:

```typescript
// Before
export function emitError(message: string, code?: string, details?: unknown): void

// After
export function emitError(message: string, code?: string, details?: unknown, context?: ErrorContext): void
```

**Text mode behavior:**
```
✗ Problem: Authentication failed
  Cause: API key is invalid, expired, or missing
  Fix: Run `jules-dispatch init` or check `JULES_API_KEY` in `.env`
```

**JSON mode behavior** (additive, backward-compatible):
```json
{
  "error": {
    "code": "AUTH_FAILED",
    "message": "Authentication failed",
    "details": null,
    "hint": "Run `jules-dispatch init` or check `JULES_API_KEY` in `.env`",
    "docsUrl": "https://github.com/..."
  }
}
```

Also add a convenience wrapper:
```typescript
export function emitTranslatedError(translated: TranslatedError, exitCode: number): never
```

This combines `translateError` → format for text/JSON → `process.exit(exitCode)`, replacing the `fail()` pattern in `cli.ts`.

### Step 3: Update catch sites in `src/cli.ts` (ERR-03)

Replace every `fail((err as Error).message, ...)` with:
```typescript
fail(translateError(err), ExitCode.GENERIC)
```

Update the `fail()` helper to accept `TranslatedError`:
```typescript
function fail(message: string | TranslatedError, code: number = ExitCode.GENERIC, errCode?: string): never {
  if (typeof message === 'string') {
    emitError(message, errCode);
  } else {
    const { problem, cause, fix, code: tCode, context } = message;
    emitError(problem, tCode, { cause, fix }, context);
  }
  process.exit(code);
}
```
